# 開発スケジュール・レビュー・WBS

## 方針

MVPは「毎日開ける記念日ダッシュボード」として、P0機能を先に完成させます。

初期開発では実ライバーデータの完全収集をブロッカーにせず、サンプルデータで次の品質を先に固めます。

- 日付計算の正確性
- 今日の記念日、近日記念日の表示
- 五十音順、デビュー日順の並び替え
- 推し登録
- 詳細ページとX共有導線
- スマホでの見やすさ

## 推奨技術構成

| 領域 | 採用案 | 理由 |
| --- | --- | --- |
| フロントエンド | Vite + React + TypeScript | 軽量で静的サイト化しやすい |
| ルーティング | React Router | `/`, `/livers`, `/livers/:id`, `/about` を明確に扱える |
| データ | 静的JSON / TypeScript data module | MVPでは管理画面なしで運用できる |
| 状態保存 | localStorage | ログインなしの推し登録に合う |
| テスト | Vitest | 日付計算ロジックを軽く検証できる |
| スタイル | CSS Modulesまたは通常CSS | 小規模MVPでは過剰なUIフレームワークを避ける |

## 開発スケジュール

想定期間は5営業日です。1人開発で、仕様確認・実装・レビューを同時に回す前提です。

| 日程 | フェーズ | 成果物 | 完了条件 |
| --- | --- | --- | --- |
| Day 1 | 土台構築 | Vite/React/TS、ルーティング、データ型、サンプルデータ | トップ、一覧、詳細の空画面が表示できる |
| Day 2 | 日付計算 | 経過日数、キリ番、周年、近日記念日の計算 | `300`, `333`, `1000`, `2434` を判定できる |
| Day 3 | 主要画面 | トップ、ライバー一覧、ライバー詳細 | P0の主要情報がサンプルデータで表示される |
| Day 4 | 操作機能 | 検索、並び替え、フィルター、推し登録、X共有 | ユーザー操作が一通り通る |
| Day 5 | 仕上げ | レスポンシブ、About、テスト、レビュー修正 | MVPとしてローカル確認できる |

## スケジュールレビュー

### 良い点

- 実データ収集とUI実装を分離しているため、開発が止まりにくい
- 日付計算を早い段階で作るため、サービス価値の中心を先に検証できる
- P1のカレンダーや通知を後回しにして、MVPの焦点が保てている
- 権利面のリスクがある画像利用を避け、名前・日付・リンク中心で始められる

### 懸念点

- 実ライバーデータの正確性確認には別途時間が必要
- 五十音順は読み仮名データがないと品質が落ちる
- 2月29日デビューや周年計算の扱いはテストで固定する必要がある
- `2434` のような特別キリ番は、今後追加しやすい設定構造が必要

### レビュー結果

スケジュールはMVP開発に問題ありません。

ただし、Day 1から実装するデータ型に `display_name_kana`、`debut_date_basis`、`source_note`、`specialDayMilestones` を入れ、後からデータ整備で詰まらない構造にします。
